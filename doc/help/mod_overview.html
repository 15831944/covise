Module Reference Guide

  • This list contains the names of all modules available with COVISE (except Test or Example modules) together with a short description and a link to the complete documentation.
  • You can get a short description for a specific module from ModuleInfo (ModuleSetup), too.
  • If you need the complete documentation for any available module, just click on the module name in the list or in the navigation in the left frame!
  • You can get the complete documentation for a specific module via 'help' in 'Further Actions', too.
  • You select your modules (usually) from the following categories
    • Color
    • Filter
    • IO_Module
    • Mapper
    • Renderer / SRenderer
    • Simulation (documentation only)
    • Tools
    • Tracer
Notes:
If you are looking for a module you might have problems starting with the right category. Strictly speaking, 'Filter' (pass only selected information) and 'Mapper' (create an image out of data) are subsets of 'Tools'; so if you don't find a suitable module in 'Filter' or 'Mapper', look up in 'Tools', and vice versa! (use the Overview given below)
The modules in the categories Renderer and SRenderer provide the same functionality - and are therefore described only once (see Renderer). The only difference between the categories shows up in collaborative work: You get  multiple instances of a Renderer module for the master and all collaborating slaves, whereas a SRenderer (S for "single") module appears only where it is created.
Documentation for obsolete Tracer modules including preprocessing modules for TetraTrace has been removed (except Tetrahedronize - kept for generating a grid of tetrahedrons).
If you have given up searching by category just try the alphabetic module list below (Complex Modules are marked with an asterisk):

Color:

ColorEdit
maps scalar values to colors on isosurfaces, cuttingplanes, vectors... (see also ColorMapEditor)

Colors
converts scalar input data into color values. It also produces Colormap output objects, which can be used as input objects to other Color modules.

Filter:

ClipInterval
clips data and grid according to a specified Interval

CuttingLine
extracts a line from a structured data set and creates a 2D object. In general you will use the module Plot to display this object.

CuttingSurface
extracts surfaces of different shape (planes, cylinders, spheres) and can also map isolines onto the plane. Linear interpolation is used to calculate the data values on the grid nodes of the resulting cutting surface.

CuttingSurfaceComp
creates a colored CuttingSurface.
The only difference between CuttingSurface and CuttingSurfaceComp is that CuttingSurfaceComp directly produces its output for a Renderer. To get a colored output you have the possibility to pass a colormap by an input port (see also ModuleGroup).
 

EliminatePolygons
eliminates polygons from a surface.

FilterCrop
allows you to reduce the number of grid lines in structured grids. It can either crop the index space of a grid by chosing min and max indices different from 0 and the initial max value, by simply omitting all grid points that lie outside this interval (this is independent for all three coordinate axes). The other possibility is to use the sample parameter to skip a certain number of intermediate grid points (this is the same values for all three directions). A sample of 2 (the default) means that every second grid point will be displayed, in the case of 3 every third grid point and so on.

IndexSurface
You can choose a dimension and an index of a structured grid for this dimension. IndexSurface calculates the surface of this area. If you work with uniform or rectilinear grids the result will be a plane.

LineReduce
replaces a straight line built up of several segment by one line segment, and thus reduces the amount of data. Two neigbouring line segments are recognized as parallel if their mutual angle (module pi) is smaller than a given limit.

SelectUsg
filters an unstructured grid according to an index field.

SimplifySurface
reduces the number of triangles in a surface.
If no data values are attached to the surface, a geometric reduction algorithm is invoked. Otherwise the reduction will be done with respect both to the data values and to the geometric properties of the surface. The module generates normals for the simplified surface.

SmoothSurface
smoothes polygonal surfaces which appear faceted or bumpy, i.e. isosurface resulting from a marching cubes algorithm.

SplitUsg

The input to this module is an unstructured grid (or set of grids) and (optional) scalar or vector data (node or element centered). It constructs output grids with the criterion that the grids at the output have a well-defined dimension, thus obtaining an object which contains all 3D elements, another with all 2D elements, and 1D and 0D respectively. The scalar or vector data is separated accordingly.


IO_Module:

GenDat
creates a chunk of a regular grid. It generates meshes based on carthesian and cyclic coordinates. It is mainly used as a generator of test and reference data.

PerformerScene
performs fake reading for Performer Models.

ReadABAQUS
reads the results from ABAQUS computations as .odb files.

ReadANSYS
reads .rst, .rfl, and .rth files, which are the result files of mechanical, FLOTRAN, or thermal analysis. In all cases READANSYS finds dynamically the degrees of freedom active in the model, from which the user may choose any option for visualisation. As for derived data, the module reads stresses and several kinds of strains for mechanical files, and the heat flux for thermal analysis.

ReadCadmould
reads in Cadmould files.

It reads in the mesh and may get information from

    • the rheology file (<basename>E0),
    • thickness of the outer layer (<basename>S##),
    • magnitudes related to the filling process (<basename>a), and
    • data from the intermediate state files (<basename>E##).
More over the module reads information from files in the CAR format.

ReadCFX
reads CFX-5 data format.

ReadDx
reads Data Explorer Ascii files - experimental version, use at your own risk; docu not yet complete.

ReadDyna3D  (SGI only - not supported on Linux)
reads the binary plot file "LS930.PTF" written by LS-DYNA3D, a software package for Nonlinear Dynamic Analysis from Livermore Software Technology Corporation. Attention: Thick shell elements are not supported!

ReadEnsight
reads data files written by Ensight (versions 5.0 and gold).

Ensight 5.0 data consists of files:

    • description of the geometry (*.geo) - required
    • result file (*.res) - optional
    • variable files  - optional
Ensight gold data consists of files:
    • description of the geometry (*.geo) - required
    • case file (*.case) - required; in this file are pointers to all other necessary files.
Timesteps are supported.

ReadEnsightNT
reads data files written by Ensight. For supported versions see table in module documentation.
Note: ReadEnsightNT is planned to replace ReadEnsight as soon as it will accept all Ensight formats.

ReadFidap
reads data from Ihs FENFLOSS.

ReadFloWorks
reads FloWorks files - experimental version, use at your own risk; docu not yet complete.

ReadFluent
reads formatted Fluent(R) files V 0.99b.

ReadIv
reads Inventor files. You have to use the Open Inventor Renderer to display the file.

ReadKiva (SGI only - not supported on Linux)
reads data sets written by  KIVA.

ReadLat
reads the ASCII latice file written by IRIS-Explorer.

ReadMovieBYU
reads MovieBYU data in ascii format. It supports scala, vector and timestep data additional to the grid data. Optional is a color file in ascii format with SGI colors.

ReadNastran
reads data written by MSC Nastran in binary output2 Version 70 format. It currently supports a subset of the NASTRAN data format.

ReadObj
reads Wavefront OBJ files.

ReadPAM (SGI only, PAM license required - not supported on Linux)
reads data sets DSY and THP files. From the DSY file it may read the grid, nodal and cell data. Global data may be read from either the DSY or the THP file.

ReadPatran
reads data sets compatible to  PATRAN 2.5 file format. It processes information of the neutral file, displacement or force result files, nodal and element result files.

ReadPlot3D  (SGI only - not supported on Linux)
reads data sets written by Plot3D.

ReadStar  (SGI and Linux - since COVISE Rel. 5.2.3)
reads data sets written by Star-CD versions 2.21 thru 3.1b. It needs a ProSTAR input file (*16, *mdl) and a result file, either stationary (*09/*pst) or transient (*29/*pstt).

ReadStl
reads Sandia Stl data.

ReadTascflowTDI  (SGI and HP only - not supported on Linux)
reads data sets written by CFX-TASCflow versions 2.7-2.10. It needs the information contained in the grid (GRD), block-off and boundary condition (BCF) and results (RSO) files.

ReadTecplot
reads Tecplot V 1.0b files.

ReadVolume
reads volume data files, and provides the scalar volume data as a uniform grid.

ReadVTF
converts the representation of a surface in ViewTech File format (specified as file browser parameter) to a format acceptable for COVISE (on the first output port); in addition, the user can choose between scalar and vector results, which are provided by the chosen file. These results can be found at the last four output ports.

RW_ASCII
reads or writes COVISE ASCII files.

RWCovise
reads or writes all data objects that are available in  COVISE.

RWCoviseGroup
reads or writes a group of COVISE files.

WriteASC
generates an ASCII file containing the information of a Covise object (accepted types are specified in the list of types of the input port). You may use this file for reporting or in case you wish to analyse the data with yout own tools.

WritePatran
writes a geometry in Patran Format - transforms a surface in COVISE format to Patran format.

WriteVolume
writes volume data files from scalar volumetric data on uniform grids.


Mapper:

IsoLines
creates isolines in a polygon. You can use a range of isovalues.

IsoSurface
creates an isosurface from a three dimensional structured or unstructured grid-based data set using the marching cubes algorithm. Optionally, data input can be mapped on the generated surface using interpolation. Each input port also accepts sets of the appropriate types.
Note: The new IsoSurface module provides the functions of the old one plus those of the former IsoSurfaceP module!

IsoSurfaceComp
creates an isosurface from  a three dimensional structured or unstructured grid-based data set using the marching cubes algorithm. Optionally, data input can be mapped on the generated surface using interpolation. This module provide all functionalities of IsoSurface but has an important advantage. You may considerably simplify your maps by using IsoSurfaceComp, because you may connect its first output port directly to a Renderer, thereby relieving you of burden of using Collect, Colors, VectorField, etc..

(IsoSurfaceP)
creates an isosurface from  data based on a three dimensional unstructured grid.
Note: The new IsoSurface module provides the functions of the  IsoSurfaceP module, too;  module IsoSurfaceP has been kept for compatibilty reasons only !

VectorField
computes lines in a vector field.
 

Renderer / SRenderer:
Note:
The modules in the categories Renderer and SRenderer provide the same functionality - and are therefore described only once. The only difference between the categories shows up in collaborative work: You get  multiple instances of a Renderer module for the master and all collaborating slaves, whereas a SRenderer (S for "single") module appears only where it is created.
(S)COVER
stands for "COVISE Virtual Environment" and is an integral part of the COVISE visualization and simulation environment. The design of COVER was developed to support technical and scientific applications. The rendering of COVER is based on IRIS Performer to make optimal use of the SGI rendering hardware.
For more information see
    • Online Help for COVER  (introduction) or
    • User's Guide, section COVER (VR Renderer) (detailed).


(S)COVER_VRML
is a version of COVER for VRML files.

(S)Plot
generates plot output for 2D objects (XMGR Plot Module).

(S)Renderer (IV Renderer)
(Open Inventor Renderer or IV Renderer for short,  also called Desktop Renderer) visualizes the output data.
For more information see

    • Online Help for IV Renderer  (introduction) or

      • User's Guide, section Renderer (Desktop Renderer) (detailed).
         (S)VRMLRenderer
         creates a VRML file from its input objects. It can be connected in the map like a normal renderer and accepts all graphical objects as input.
 

Simulation:

    not part of this Release - information only

StarCD

starts a STAR simulation on the same machine or a remote one and establishs a TCP connection to the simulation process. Via this connection the module can set boundary conditions to the simulation and can receive data from the simulation. The exchange of data and boundary conditions is implemented in user subroutines which have to be linked with the STAR simulation code.


Tools

    AssembleUsg
    converts blocks of unstructured grids (per timestep) to one unstructured grid (per timestep) without any block information. This module has been developed to use the module TetraTrace also for blocks of grids.

    BlockCollect
    collects different objects and puts them into one set.
    You can choose among three different rules to build up this set:

      1. Set of Elements
      2. Cat Timesteps
      3. Merge Blocks


    BoundingBox
    writes a message with information about the smallest parallelepiped parallel to the globale coordinate system which contains an unstructured grid (or set of grids).

    Calc
    allows you to perform mathematical calculations with scalar and/or vector objects. The result may be a scalar or a vector. The module interprets a string according to a syntax described below.

    CellToVert
    interpolates per-cell data to per-vertex data
    (Simulation codes like Fluent or StarCD compute results per-cell. Most COVISE modules expect per-vertex data.)

    CheckObj
    tests the object received at the input port (unstructured grid, polygons, lines, triangle strips).

    Collect
    combines geometric primitive objects like points, lines, polygons, triangle strips with colors, normals and textures.

    CropUsg
    crops the parts of an unstructured grid which are outside of a bounding box.

    CutGeometry
    cuts geometry objects. The part which should be cut away is defined through a plane. This is usually used to make the inner parts of geometry objects visible.

    CuttingSurface3DTex
    combines a cuttingsurface and colors on a cuttingsurface with an uniform grid and color on a uniform grid into a set of two geometry objects. The uniform grid and the colors on the uniform grid represent the voxels of a 3D texture.

    DerivOperators
    works out gradients of scalar fields, or the divergence or curl of vector fields.

    DisplaceUsg
    displaces an unstructured grid by using displacement vectors for every vertex.

    DomainSurface
    computes the outer surface or edges of an unstructured grid.

    DomainSurfaceStr
    computes the outer surface or edges of a structured grid.

    FixPart
    fixes one part in a simulation over timesteps and moves all other parts relativ to the fixed one.

    FixUsg
    removes double and unused vertices.

    GenNormals
    generates normals for a set of polygons.

    GetSetelem
    performs the operation of selecting an element out of a set.

    GridSurface
    analyzes grids and generates surfaces as polygons. It shows all surfaces,  i.e. including the inner ones.

    Histogram
    generates a histogram from a data set, i.e. counts for every data value how often it appears in the data set.

    ImageToTexture
    generates a texture for a polygon input object given a TIFF file. The default mapping that relates the geometry point
    coordinates with the texture coordinates is a simple projection onto the XY plane. You may introduce your own mapping if the module
    gets a vector field as input object.

    Interpolate
    creates a set of unstructured vector or scalar data by interpolation of the incoming data. For each point of the data set the intervall between zero and the current value is decomposed into a given number of segments.

    MakeOctTree
    generates octtrees for unstructured grids and polygons.

    MakePlots
    generates a set of DO_Unstructured_2D_Data objects, which may be  plotted by Plot (see Renderer modules) out of some scalar fields.

    MakeTransient
    allows you to generate a specified number of timesteps (default 72) out of 'static' data. With a Timestep Sequencer that shows up in the Renderer you can go thru your timesteps, either step by step or continuously using 'play'.

    Material
    contains a list of different color values from which the user can select.

    MinMax
    calculates the minimum and maximum value of a scalar dataset. It outputs either a min-max pair that can be used for example as input for a colormap module, or a list of x-y pairs that shows the distribution of values over the range.
    (has been moved from "Filter" to "Tools")

    PartSelect
    is able to manage sets to cull parts of the geometry or data based on the part attribute.

    PipelineCollect
    reduces the maximum usage of memory by executing a pipeline stepwise.

    Sample
    transforms an unstructured grid and data on the unstructured grid into a uniform grid and data on the uniform grid. As the uniform grid has the extension of the bounding box around the unstructured grid, data on parts of the uniform grid which are outside the unstructured grid are either set to a certain value or max_float. The module Colors computes a transparent color for MAX_FLT data so those parts appear transparent in the Renderer.
    A uniform grid and colors on the uniform grid can also be seen as a 3D texture. If you cut a plane out of the 3D you get a 2D texture (an image). Mapping this texture on a plane gives similar information as a cuttingsurface through the 3D grid with data mapped on the surface as colors.
    3D textures are supported by Octane and Onyx systems and are computed by the graphics hardware and are thus very fast.

    Scalar2Vector
    combines scalar data given at the input ports to vector data. If input data have attributes they are summarized where it makes sense.

    ShowFaceNormal
    computes the normal of a polygon or a triangle strip.

    ShowGrid
    for structured grids: computes the grid lines of the grid.
    There are different display methods:

    • show all grid lines
    • show the grid lines on the outer surface (hull) only
    • show three sides of the hull only (the user may choose among 8 options that determine which three sides are selected)
    • show only the bounding box only
    • show the edge lines only
    for unstructured grids: displays only one cell; the cell is selected through the parameter position.

    Solutions
    calculates Plot3D solution data (data read by ReadPlot3D).

    Sphere
    builds spheres (made up of triangle-strips) out of points (e. g. retrieved using a particle tracer).

    StoU
    transforms a rectiliniar, structured or an uniform grid received to the inout port into an unstructured grid which can be composed of tetrahedrons, hexahedrons, pyramids or prisms.

    TensorScal
    extracts invariants from tensor fields.

    Tetrahedronize
    converts unstructured mixed-cell-type grids into tetrahedra grids.

    Transform
    manipulates grids and geometry objects. Transform makes it possible to mirror, translate, scale, rotate, and tile objects and their data.

    Tube
    creates cylinders around lines. The shape of the tubes depends on the radius, number of sides (scalar parameters) and the type of the cylinder (open, closed or arrow).

    VectorScal
    extracts scalar values from vector fields.


Tracer:

Tracer
performs particle tracing within vector fields inside any kind of grids. It can either show Floating Particles or connect these to Streamlines, Streaklines, and Pathlines, which are different for transient data.

TracerComp
performs particle tracing within vector fields inside any kind of grids,. It can either show Floating Particles or connect these to Streamlines, Streaklines, and Pathlines, which are different for transient data. The key difference with respect to the normal Tracer module is that TracerComp relieves you of the burden of connecting the output with Colors, Collect; and Sphere modules. You may directly connect the first output port to the Renderer.
